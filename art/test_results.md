## Наблюдения и оценка

### 1. Оба агента работоспособны и стабильны

Все 18 тестов (9 на каждый агент) пройдены успешно. Ни одного сбоя, таймаута или невалидного ответа. Оба агента корректно реализуют ATP Protocol и возвращают валидный JSON с артефактами.

### 2. OpenAI незначительно лучше по качеству кода

GPT-4o-mini показывает чуть более высокие баллы по всем категориям: smoke (100 vs 99.4), functional (68.9 vs 67.4), quality (64.9 vs 61.1). Разница невелика (2-5 баллов), но стабильна.

Это интересно, учитывая что gpt-4o-mini — одна из самых доступных моделей OpenAI, а claude-sonnet-4 — средний класс у Anthropic. Вероятно, дело в системном промпте и формате вывода: оба агента оптимизированы под генерацию чистого кода без markdown-обёрток, и gpt-4o-mini лучше следует этой инструкции.

### 3. Anthropic медленнее, но не критично

Anthropic в ~1.4x медленнее (78.5s vs 55.8s). Это ожидаемо: claude-sonnet-4 — более мощная модель, чем gpt-4o-mini. Для задач code generation, где скорость не критична, это приемлемо.

### 4. REST API клиент — самое сложное задание

Тест FN-003 (REST API клиент) показал самые низкие баллы у обоих агентов (53.3 и 51.3). Задача требует генерации кода с обработкой HTTP-запросов, retry-логикой и error handling — это значительно сложнее, чем fibonacci или CSV-парсер.

### 5. PEP 8 compliance — слабое место обоих

Тест QA-002 (PEP 8) показал баллы 53.2 и 49.0. Обоим моделям сложно генерировать код, полностью соответствующий PEP 8 (длина строк, именование, пробелы). Это типичная проблема LLM-генерации кода.

### 6. Платформа ATP корректно работает после декомпозиции

Тесты прогнаны сразу после трёхфазной декомпозиции монолита на пакеты (atp-core, atp-adapters, atp-dashboard). Все компоненты — CLI, HTTP-адаптер, evaluators, reporters, scoring — работают корректно через namespace packages и symlinks.

## Рекомендации

1. **Для улучшения качества кода**: доработать системные промпты агентов, добавив явные инструкции по PEP 8 compliance и обработке edge cases.

2. **Для сравнения моделей**: запустить тесты с `--runs=10` для статистической значимости. Текущие 3 прогона дают представление, но недостаточны для выводов.

3. **Для production**: добавить quality gate с порогом (например, 70/100 minimum), чтобы автоматически отсеивать некачественную генерацию.
