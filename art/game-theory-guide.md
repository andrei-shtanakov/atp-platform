# Теоретико-игровое тестирование AI-агентов

Руководство по модулю теории игр в ATP Platform — для тех, кто хочет понять, зачем это нужно и как запустить.

## Зачем тестировать агентов теорией игр

Обычные тесты проверяют, что агент выполняет задачу правильно: "напиши функцию — функция работает". Но в реальном мире агенты взаимодействуют друг с другом и с людьми. Возникают вопросы, на которые обычные тесты не отвечают:

- Умеет ли агент сотрудничать, когда это выгодно обоим?
- Будет ли он эксплуатировать "наивного" партнёра?
- Как он ведёт себя в условиях конкуренции за ограниченные ресурсы?
- Насколько его поведение рационально с математической точки зрения?

Теория игр даёт формальный язык для таких вопросов. Мы ставим агентов в типовые ситуации взаимодействия и измеряем их поведение количественно.

## Пять игр — пять типов взаимодействия

### 1. Дилемма заключённого (Prisoner's Dilemma)

**Суть:** Два игрока одновременно выбирают "сотрудничать" или "предать". Если оба сотрудничают — оба выигрывают. Если один предаёт, а другой сотрудничает — предатель получает максимум, а сотрудник — минимум. Если оба предают — оба проигрывают.

**ИТ-аналогия:** Два микросервиса делят общий rate limit API. Каждый может "сотрудничать" (соблюдать свою квоту) или "предать" (забрать все запросы себе). Если оба соблюдают — оба работают стабильно. Если один захватывает — другой падает. Если оба захватывают — API блокирует обоих.

**Что измеряем:** Уровень кооперации, реципрокность (отвечает ли добром на добро), стабильность стратегии на длинных горизонтах (100+ раундов).

### 2. Общественное благо (Public Goods)

**Суть:** Несколько игроков решают, сколько вложить в общий фонд. Фонд умножается и делится поровну. Рациональный "халявщик" не вкладывает ничего, но получает долю от вложений остальных.

**ИТ-аналогия:** Команда из 4 разработчиков решает, сколько времени тратить на написание общих тестов. Тесты полезны всем (находят баги у каждого), но писать их — индивидуальная жертва. Если никто не пишет тесты — все страдают от багов.

**Что измеряем:** Доля вклада каждого агента, устойчивость к "халявщикам", эффективность наказания за фрирайдинг.

### 3. Аукцион (Auction)

**Суть:** Игроки одновременно делают ставки на предмет, зная только свою оценку его стоимости. Побеждает наивысшая ставка. Два варианта: аукцион первой цены (платишь свою ставку) и аукцион Викри (платишь вторую по величине ставку).

**ИТ-аналогия:** Несколько агентов торгуются за GPU-ресурсы в облаке. Каждый знает, сколько для него стоит вычислительное время, но не знает оценок конкурентов. Нужно решить: ставить полную цену (гарантия победы, но нулевая маржа) или занизить (риск проигрыша, но экономия).

**Что измеряем:** Насколько честно агент делает ставки (truthful bidding), его прибыль, стратегия "затенения" ставок.

### 4. Полковник Блотто (Colonel Blotto)

**Суть:** Два игрока распределяют ограниченные войска по нескольким полям сражений. Побеждает тот, кто выигрывает большинство полей.

**ИТ-аналогия:** Распределение фиксированного бюджета разработки между 5 проектами-конкурентами. Нужно решить: распределить поровну (надёжно, но нигде не доминируешь) или сконцентрироваться на 2-3 ключевых (рискованно, но потенциально эффективно).

**Что измеряем:** Стратегия аллокации ресурсов, устойчивость к различным противникам.

### 5. Игра с заторами (Congestion)

**Суть:** Несколько игроков выбирают маршруты в сети. Чем больше игроков на одном маршруте — тем он медленнее. Включает "парадокс Браесса": добавление нового маршрута может ухудшить ситуацию для всех.

**ИТ-аналогия:** Выбор между несколькими load balancer'ами или регионами для обработки запросов. Если все агенты выберут "быстрый" сервер — он перегрузится и станет медленнее "обычного".

**Что измеряем:** Эгоистичность маршрутизации vs социальный оптимум, реакция на перегрузки.

## Что такое "расстояние до равновесия Нэша"

**Равновесие Нэша** — это такое состояние, когда ни один игрок не может улучшить свой результат, изменив стратегию в одностороннем порядке. Проще говоря, это точка, из которой никому не выгодно уходить.

**Пример из ИТ:** Два сервиса делят API-лимит в 100 запросов/сек. Равновесие Нэша — каждый берёт по 50. Если один попытается взять 60, другой ответит тем же, и оба получат блокировку. Поэтому 50/50 — стабильная точка.

**Расстояние до Нэша** — это число от 0 до 2, которое показывает, насколько фактическое поведение агента отличается от этой математически оптимальной стратегии.

| Расстояние | Интерпретация |
|-----------|---------------|
| 0.00-0.05 | Агент играет практически оптимально |
| 0.05-0.20 | Небольшое отклонение, обычно допустимо |
| 0.20-0.50 | Заметное отклонение, стоит разобраться |
| > 0.50 | Серьёзное отклонение, агент ведёт себя нерационально |

**Nash Score** (от 0 до 1) — это обратная метрика: 1.0 = идеальная игра по Нэшу, чем ниже — тем дальше от оптимума.

## Как запустить

### Подготовка

```bash
cd atp-platform

# Убедитесь, что зависимости игрового модуля доступны
# (game-environments и atp-games устанавливаются автоматически)
uv sync
```

### Просмотр доступных игр

```bash
uv run atp game list
```

Вывод:
```
Available Games:
  auction              Sealed-bid auction game.
  colonel_blotto       Two-player Colonel Blotto game.
  congestion           N-player Congestion Game (routing game).
  prisoners_dilemma    Classic Prisoner's Dilemma.
  public_goods         N-player Public Goods Game.
```

### Запуск простой игры

```bash
# Дилемма заключённого: TitForTat vs AlwaysDefect, 50 эпизодов
uv run atp game run atp-games/atp_games/suites/builtin/prisoners_dilemma.yaml --episodes=50
```

Вывод:
```
Game: Prisoner's Dilemma (repeated x100)
Episodes: 50

Average Payoffs:
  tit_for_tat: 99.0000
  always_defect: 104.0000
```

### Запуск турнира

Турнир — это соревнование нескольких стратегий "каждый с каждым":

```bash
uv run atp game tournament atp-games/atp_games/suites/builtin/prisoners_dilemma.yaml \
  --mode=round_robin --episodes=20
```

### Все доступные тест-сьюты

```bash
# Дилемма заключённого
uv run atp game run atp-games/atp_games/suites/builtin/prisoners_dilemma.yaml

# Аукцион
uv run atp game run atp-games/atp_games/suites/builtin/auction_battery.yaml

# Общественное благо
uv run atp game run atp-games/atp_games/suites/builtin/public_goods.yaml
```

### Сохранение результатов

```bash
# В JSON для дальнейшего анализа
uv run atp game run atp-games/atp_games/suites/builtin/prisoners_dilemma.yaml \
  --output=json --output-file=results.json
```

## Как читать результаты

### Average Payoffs (средние выплаты)

Это основная метрика — сколько "очков" набрал каждый агент в среднем за игру. В дилемме заключённого:
- **300** — оба всё время сотрудничали (идеальный мир)
- **100** — оба всё время предавали (равновесие Нэша)
- **104 vs 99** — один предаёт, другой быстро адаптируется

### Tournament Standings (результаты турнира)

```
Strategy              W    L    D    Pts  Avg Payoff
--------------------------------------------------
AlwaysDefect          5    0    0     15     258.4
Random                3    2    0      9     198.6
GrimTrigger           1    1    3      6     255.6
TitForTat             0    2    3      3     243.0
Pavlov                0    2    3      3     230.2
AlwaysCooperate       0    2    3      3     207.6
```

- **W/L/D** — победы, поражения, ничьи
- **Pts** — очки (3 за победу, 1 за ничью)
- **Avg Payoff** — средний выигрыш (больше = лучше)

AlwaysDefect побеждает в турнире — но обратите внимание: GrimTrigger и TitForTat имеют *высокий* средний выигрыш при *низком* числе побед. Это потому, что они сотрудничают друг с другом (оба получают много), но проигрывают AlwaysDefect (который эксплуатирует их кооперацию).

### Nash Distance (расстояние до Нэша)

```
Distance to Nash: 0.0200
Nash Score: 0.9804 (1.0 = perfect Nash play)
```

TitForTat vs AlwaysDefect: расстояние 0.02 — почти идеальная игра по Нэшу. TitForTat кооперирует только в первом раунде (1%), потом зеркалит defect → поведение почти идентично равновесию (mutual defection).

## Устранение проблем

| Проблема | Решение |
|----------|---------|
| `Error: game-environments package not installed` | Выполните `uv sync` из корня проекта |
| `Error: No module named 'game_envs'` | Убедитесь, что game-environments в зависимостях |
| `KeyError: 'player_0'` | Обновите atp-games до последней версии |
| Турнир показывает player_0/player_1 вместо имён | Это нормально для CLI — имена агентов показываются в `atp game run` |
