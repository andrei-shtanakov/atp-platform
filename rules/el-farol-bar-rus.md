# El Farol Bar — правила (ATP Platform)

Документ собран из реализации игры в репозитории. Идентификатор в реестре: `el_farol`, класс: `ElFarolBar`, файл: `game-environments/game_envs/games/el_farol.py`.

---

## Идея

Задача бара Эль-Фароля (Brian Arthur, 1994): N игроков независимо выбирают временные слоты посещения. Если число посетителей в слоте достигает или превышает порог вместимости, слот считается **переполненным (crowded)** — пребывание там нежелательно. Игроки стремятся проводить больше времени в непереполненных слотах.

В реализации это повторяющаяся игра с **одновременными** ходами (`MoveOrder.SIMULTANEOUS`), один шаг фреймворка = один **день**.

---

## Игроки и масштаб

- Число игроков `num_players`: от **2** до **1000** (по умолчанию в конфиге — 100).
- Идентификаторы игроков: `player_0`, …, `player_{N-1}`.

---

## Действие

- Каждый день каждый игрок выбирает **список индексов слотов** — целые числа в диапазоне `[0, num_slots - 1]`.
- Слоты в списке должны быть **уникальными**.
- Не более **`MAX_SLOTS_PER_DAY = 8`** слотов за день.
- Пустой список `[]` означает «остаться дома» в этот день (ни один слот не посещён).
- Некорректный ввод приводится к безопасному списку через `sanitize` (дубликаты, выход за диапазон, не-списки и т.д.).

По умолчанию в конфиге: `num_slots = 16`, `slot_duration = 0.5` ч — то есть 16 полчасовых слотов на день (8 часов суммарно), если не менять конфиг.

---

## Переполнение слота

Для каждого слота считается **число игроков**, выбравших этот слот в текущий день.

- Слот **не переполнен (happy)** для игрока, если занятость слота **строго меньше** `capacity_threshold`.
- Слот **переполнен (crowded)** для игрока, если занятость **≥** `capacity_threshold`.

По умолчанию `capacity_threshold = 60` (в примере сьюта ниже используется меньшее значение для малого N).

---

## Выплата за день (per-round payoff)

Для каждого игрока в день:

- `happy` — число посещённых им слотов, где занятость `< threshold`.
- `crowded` — число посещённых им слотов, где занятость `≥ threshold`.

**Выплата за этот день:** `happy - crowded` (чистое «хорошее минус плохое» по слотам).

Накопительно ведутся `t_happy` и `t_crowded` (в **слотах**, не в часах).

---

## Финальный счёт (`get_payoffs`)

После всех дней:

- Если суммарные часы посещения `(t_happy + t_crowded) * slot_duration` **строго меньше** `min_total_hours`, итоговый payoff игрока = **0** (дисквалификация).
- Иначе: **`t_happy / max(t_crowded, 0.1)`**.

По умолчанию `min_total_hours = 0.0` — дисквалификация по минимуму часов отключена, если явно не задать порог.

---

## Информация у игрока (наблюдение)

В `game_state` игрок видит в том числе:

- историю посещаемости по прошлым дням (`attendance_history`) — **публичная** информация;
- `capacity_threshold`, `num_slots`, `slot_duration`, `min_total_hours`;
- свои накопленные `your_t_happy_slots`, `your_t_crowded_slots`, `your_total_hours`.

Текущие планы других в этот день **не** раскрываются (как в docstring / промпте: видна только прошлая посещаемость).

---

## Параметры конфигурации (`ElFarolConfig`)

| Поле | Смысл | Значение по умолчанию |
|------|--------|------------------------|
| `num_players` | число игроков | 100 |
| `num_rounds` | число дней | 30 |
| `num_slots` | слотов в день | 16 |
| `capacity_threshold` | порог «толпы» в слоте | 60 |
| `min_total_hours` | минимум часов, иначе 0 в конце | 0.0 |
| `slot_duration` | длительность слота (часы) | 0.5 |

Наследуются общие поля `GameConfig` (например `discount_factor`, `noise`, `seed`), если используются в вашем сценарии.

---

## Заметки из кода (теория)

- Симметричная смешанная Нэш: вероятность посетить данный слот `p* ≈ threshold / num_players` (в промпте игры это сформулировано явно).
- Гетерогенные стратегии в популяции часто дают лучший совокупный исход, чем эта Нэш, в агрегате (как в docstring класса).
